# Repository Guidelines

## Project Structure & Module Organization
- `src/pages/` contains Astro routes (one file per page).
- `src/components/` holds reusable UI components.
- `src/layouts/` defines shared page layouts.
- `src/content/` stores content collections: `posts/`, `thoughts/`, `pages/`, with schema in `src/content/config.ts`.
- `public/` is for static assets served as-is (images, icons).
- `dist/` is the production build output (generated by Astro).
- Key config lives in `astro.config.mjs`, `tailwind.config.mjs`, and `src/config.ts` (site metadata/navigation).

## Build, Test, and Development Commands
- `npm install` installs dependencies.
- `npm run dev` starts the local Astro dev server.
- `npm run build` creates a production build in `dist/`.
- `npm run preview` serves the production build locally.

## Coding Style & Naming Conventions
- Use 2-space indentation (match existing `.ts`/`.astro` files).
- Prefer descriptive, lowerCamelCase for variables and functions.
- Components and layouts use PascalCase filenames.
- Content entries live under the matching collection folder and use kebab-case filenames when possible.
- No formatter/linter is configured; keep changes consistent with surrounding code.

## Testing Guidelines
- No automated test framework is configured in this repo.
- If adding tests, place them under a new `tests/` directory and document the command in `package.json`.

## Commit & Pull Request Guidelines
- Commits follow a Conventional Commits style in history (e.g., `feat: ...`, sometimes with a full-width colon `feat：...`).
- Use clear, scoped messages: `feat: add archive pagination` or `fix: correct feed url`.
- PRs should describe the change, list any user-facing impact, and include screenshots for UI updates.

## Content & Configuration Notes
- Update site metadata and navigation in `src/config.ts`.
- Content frontmatter should match the existing collection schemas (date formats like `YYYY-MM-DD`).
